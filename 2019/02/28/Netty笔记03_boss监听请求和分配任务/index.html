Netty笔记03_boss监听请求和分配任务

发表于 2019-02-28

经过前面两章，boss已经启动及处理监听工作之中，那么本章分析监听及分配任务的细节。

类图

在绑定注册时，Channel会关联一个NioEventLoop，这个EventLoop就是属于boss的，用于监听请求。

EventLoopGroup负责为每个新创建的Channel分配一个EventLoop。

一旦一个Channel被分配给一个EventLoop，它将在它的整个生命周期找那个都使用这个EventLoop。

关于selectStrategy.calculateStrategy(selectNowSupplier, hasTasks())

这行代码的作用，

1）当有任务时，直接调用selectNowSupplier.get()方法，马上进行一次selectNow()操作，返回已经准备好的通道数量（>=0）。

2）当没有任务时，直接返回-1，进入select(…)方法。

这样就保证了有任务时，能够及时执行任务，因为select是阻塞的，而selectNow是非阻塞的。

关于wakenUp

这个是一个AtomicBoolean类型，由于这里的Selector.select是会被阻塞的（在一定的时间内会一直尝试获取请求，超时后则跳出）。在被阻塞时，如果有新的任务到来，用这个变量强制唤醒。

小结

这个方法是所有boss和worker的核心方法，可以分为三个部分：

1）select，如果是boss，会监听ACCEPT；而worker，监听READ。

2）IO任务，如果是boss，发现有请求ACCEPT后，将会创建一个新的Channel，调用register方法，将worker注册进去；如果是worker，则监听到READ后，处理Handler等。

3）其他任务。

select()方法

这个方法的作用，当有任务来时，将执行selectNow()，或者唤醒select()，使得其不会阻塞，更快地执行这个任务。当没有任务时，会在一定的时间内，不断地执行阻塞的select()方法，监听请求。

首先需要了解Nio的一些情况，如果先执行了selector.wakeup()，那么接下来的select.select(time)将不会阻塞，而是马上返回；如果先执行了selector.select(time)，在其阻塞时间内，再执行selector.wakeup()，那么selector将会马上返回。

OK，正式进入分析，

情况一，run()传入false，进入select()方法，期间也没有其他新的任务到来，那么一直会循环执行`select(timeMillis)阻塞，最后跳出，这个是最常见的情况。

情况二，run()传入false，进入select()方法，期间有新的任务到来，执行了4）这段代码，导致wakenUp变为true，比企鹅唤醒了selector。在select()中，进入2）这段代码，wakenUp.compareAndSet(false, true)是false，然后往下，继续selector.select(timeoutMillis);，此时不会阻塞，马上返回。接着，进入if(…wakenUp.get()…)判断，brak，跳出。回到run()方法的1）后，由于if(wakenUp.get())，所以执行selector唤醒操作。

情况三，和情况二类似，不过存在wakeUp竞争。run()传入false，进入select()方法。同时有新的任务到来，执行到4）这里，但是wakenUp还没来得及设置为true。而在select()方法中，执行到2）抢先将wakenUp设置为true，继而执行selectNow()，而且在finally中唤醒selector。最后也是进入if(…wakenUp.get()…)判断，跳出。回到run()方法的1）后，由于if(wakenUp.get())，所以执行selector唤醒操作。

为什么在run()方法的1）中，还需要增加唤醒selector的动作？这是用于在run()方法的下一个循环中，唤醒selector的。

在二和三这些情况下，wakenUp是true，并且做了selector唤醒。然后，进入下一个循环。

情况四，run()传入ture，进入select()方法，区间没有其他新的任务到来。那么进入4）后，会执行select(timeMillis)，但是由于在上一个循环中执行了唤醒操作，所以不会阻塞，而是马上返回。继续，在4）中判断if(…oldWakenUp…)，所以会跳出。回到run()方法的1）后，不执行selector唤醒操作。

情况五，run()传入true（oldWakenUp为true，wakenUp为false），进入select()方法，期间有新的任务到来，执行了4），wakenUp设置为true，再次执行selector唤醒操作（和情况二、三重复唤醒）。在select()方法中，继续2），执行select(timeoutMillis)，然后跳出。

类似地，还有其他一些情况，由此可见selector能够在新任务来时，不进行阻塞，而是马上唤醒。

总结

以下没有指明类名的方法，均是属于NioEventLoop。

1）boss进入select()，监听ACCEPT，发现有请求连接，此时selectedKeys被修改。

2）boss进入processSelectedKeys()，发现selectedKeys被修改，并且值readOps()的值为ACCEPT。

3）boss进入AbstractNioMessageChannel$NioMessageUnsafe#read()方法，创建一个新的NioSocketChannel。

4）boss进入DefaultChannelPipeline#fireChannelRead()方法，该pipeline对$ServerBootstrapAcceptor -> tail。

5）boss进入ServerBootstrapAcceptor#channelRead()，这个acceptor就是真正分配任务的地方，过程这样分为两步：

5.1）得到第3）步中创建的NioSocketChannel，并且调用NioSocketChannel.pipeline().addLast(childHandler)，添加childHandler。这里的childHandler，就是在boss和worker初始化时，调用ServerBootstap..childHandler(…)添加进来的。

5.2）从childGroup在选出一个worker，调用AbstractChannel$AbstractUnsafe#register()方法，向worker添加注册任务。至此，boss的监听及分配任务的工作完成。
