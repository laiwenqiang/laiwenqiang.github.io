用到了两个if (singleton == null)，顾名思义，这种方法就称作双检锁。

常规思路

一开始我认为是线程安全的。

  1. 假设有线程A和B，他们执行到第6行。线程A先占有锁，线程B挂起等待。
  2. 线程A创建对象。
  3. 线程B被唤醒，发现对象已被创建，不会再次创建对象。

这个难道不是线程安全的吗？

深入思考

jvm创建对象并非是原子性的，导致下面的流程可能发生：

  1. 线程A获取锁，执行到第8行，开始创建对象。
  2. 同时，线程B执行到第5行，判断对象是否为null。
  3. 线程Bif (singleton == null),判断结果为false，返回对象。
  4. 然而，此时线程A创建对象并未完全结束，造成线程B获取到的对象出现问题。

创建对象的步骤

由于new一个对象并非原子性的，它是分为好几个步骤，所以导致上面的第4步出现问题。

  1. 分配对象的内存空间。
  2. 设置实例（singleton）指向内存空间（此时singleton不为null，可是对象还未初始化完成）。
  3. 初始化对象。

jvm加载类的步骤

  1. 加载阶段（load）：

    从硬盘上寻找对应的class文件，将读取二级制数据加载到内存，放在运行时数据区的方法区中。在堆区创建一个java.lang.Class对象，用来封装在方法区内的数据结构。

  2. 连接阶段（link）：

    2.1 验证：检查数据是否符合规范。

    2.2 准备：主要为类的静态变量分配内存，并设置默认的初始值。

    2.3 解析：查找常量池中的类、字段、方法、接口的符号引用，将他们替 换为直接引用。

  3. 初始化（init）：

    为类的静态变量设置正确的初始值。

可见，由于创建一个对象并非原子性，导致该段代码出现问题。

线程通信之Condition条件变量

发表于 2017-07-16 | 分类于 java基础

说到Lock就不得不说Condition条件变量，简单的说，作用类似于synchronized的wait和notify。但是，功能会更加强大一些。本文主要从Condition的用法着手，讲解一个例子。

场景

  1. 三个线程，线程A、B和C，

  2. A打印“01”，B打印“02”，C打印“03”，

  3. 并且按照顺序循环打印。

分析

这里的主要难点有两个，

  1. 按照顺序打印。

    设置三个条件变量（condition），由前一个线程唤醒后一个线程，如此一来即可实现顺序打印。

  2. 前一个线程在执行完后，会唤醒后一个线程。在前一个线程唤醒后一个线程的时候，后一个线程可能还未处于的等待唤醒的状态。

    设置三个标志（flag），如果为true，表示当前任务处于准备好的状态，可以执行；否则表示为需要被前一个线程唤醒。
